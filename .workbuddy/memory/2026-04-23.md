## 合同编辑页"已选地块"未加载（根因深挖）
- 表面现象：编辑合同16时"已选地块"显示为空
- 实际根因：`add_contract()` 用 `SCOPE_IDENTITY()` 获取新合同ID不稳定，返回 None 后 `if contract_id:` 跳过了 ContractPlot 插入
- 证据：合同13-16 的 `plot_count` 均为0，而合同4-12正常有地块关联
- 修复：`add_contract()` 改用 `OUTPUT INSERTED.ContractID` 在 INSERT 语句中直接获取新ID，并保留 SCOPE_IDENTITY 作为回退
- 现有坏数据（合同13-16无 ContractPlot）需用户在编辑页面重新选择地块并保存来修复
- 前端渲染时序也做了优化：`renderSelectedPlots()` + `calculateTotal()` 在 `loadContractData()` 填充 selectedPlots 后立即调用

## 合同创建未联动应收（合同16）
- **问题**：合同编号 ZTHYHT1120260423026（ID=16）创建后没有插入应收记录
- **根因**：合同16创建时 `SCOPE_IDENTITY()` 返回 None（当时 OUTPUT INSERTED 修复可能尚未部署），`if contract_id:` 分支跳过 → ContractPlot 和应收均未创建。后来用户编辑合同（13:20），`update_contract` 重建了 ContractPlot，但 `update_contract` 之前没有应收同步逻辑
- **修复1**：为合同16手动补创建应收记录（ReceivableID=43，金额22915.44）
- **修复2**：`update_contract()` 新增应收同步逻辑：
  - 已有应收 → 更新金额/描述/RemainingAmount（扣减已付金额）
  - 无应收但金额>0 → 创建新应收
- **修复3**：`add_contract()` 和 `update_contract()` 对 `rent_adjust` 参数增加 `float()` 类型保护，防止字符串类型导致 `total_rent + rent_adjust` TypeError
- **修复4**：路由层 `contract.py` 的 add/edit 也对 `rent_adjust` 做 `float()` 转换
- **修复5**：`add_contract()` 增加详细日志输出（contract_id/total_rent/rent_adjust/actual_amount/expense_type_id/应收检查结果），方便排查
- **备注**：合同13-15 有应收但无 ContractPlot（历史数据问题），暂不处理
