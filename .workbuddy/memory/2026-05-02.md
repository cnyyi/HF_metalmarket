## AI Agent 修复（P0+P1 部分完成）

### P0 安全修复 ✅
1. **MerchantID 校验漏洞** → 重写 `_validate_merchant_condition()`，正则校验 WHERE/ON 子句中必须包含 `MerchantID = {merchant_id}`，拒绝仅在 SELECT/ORDER BY 中出现的情况，支持别名形式
2. **API Key 泄露** → `_call_llm()` 异常时不再返回 `str(e)` 到前端，只返回通用提示，详情写日志
3. **XSS 防护** → chat.html 和 wx_chat.html 引入 DOMPurify，所有 `marked.parse()` 输出经 `DOMPurify.sanitize()` 过滤
4. **SQL 注释绕过** → `_remove_comments()` 用空格替代注释（防关键字粘连），分号检测后容忍单条语句末尾分号

### P1 可靠性修复 ✅
5. **OpenAI 客户端复用** → `AgentService.__init__` 初始化，`_get_client()` 懒加载单例，配置变更时重建
6. **历史消息重复** → 删除 `chat()` 中手动 append 用户消息的重复行（_load_history 已包含）
7. **Schema 同步** → 重写 DATABASE_SCHEMA，对齐实际数据库：Receivable.Status=未付款/IsActive=1 软删除/Contract 增加金额字段/SalaryProfile 补全字段/DormRoom/DormOccupancy/DormBill 字段修正/删除 ExpenseType 表/增加 AgentConversation/AgentMessage 等
8. **Few-shot 示例** → 增加 6 个典型 Q&A 示例（应收/空闲地块/合同到期/水电费/商户应收/合同统计），包含 IsActive=1、Sys_Dictionary JOIN 等关键模式
9. **SQL 自动修复** → `_execute_sql()` 返回 `(result, error)` 元组，失败时调 `_retry_sql_fix()` 将错误信息反馈给 LLM 修正一次
10. **简单结果优化** → 结果 ≤3 行且 <500 字符时跳过第二次 LLM 调用，直接格式化展示

### Schema 枚举值动态化 ✅
11. **根因**：Prompt 手写 Contract.Status=N'执行中'，实际数据库存的是 N'有效'/N'生效'，导致查询返回 0
12. **数据实证**：
    - Contract.Status: '有效'(20), '生效'(9) — 不是'执行中'/'已到期'/'已终止'/'待签约'
    - Merchant.MerchantType: 'company'/'业务往来'/'个体工商户'/'公司'/'意向客户' — 不是'企业'/'个体'
    - Merchant.Status: '1'(3), '正常'(25) — 有脏数据 '1'
    - Account.AccountType: 'Bank'/'WeChat' — 不是'现金'/'银行'/'微信'
    - Plot.Status: '已出租'(36), '空闲'(15), None(2) — 无'维修'
13. **即时修复**：所有枚举值改为数据库实际值，Few-shot 示例同步修正
14. **长效机制**：新增 `_refresh_enum_cache()` 从数据库动态查询 DISTINCT 值，缓存 1 小时自动刷新；`build_system_prompt()` 用 `_get_enum_hint()` 动态替换 Schema 中的枚举值；即使硬编码值过时，数据库查到的实际值会覆盖
