## 1. 模块概述

### 1.1 业务背景

市场运营中存在大量固定/周期性支出，需要记录详细信息并归集管理。现有应付账款（Payable）模块只能记录单笔费用，无法满足以下业务场景：

- **一次业务含多个费用项**：如垃圾清运包含运费+处置费，拆分记录后无法看出是同一次业务
- **需要记录明细描述**：如临时工清扫，需要记录「哪天、谁、干了什么、多少钱」，现有应付只有一行备注
- **按大类归集统计**：想知道「垃圾清运」这个大类总共花了多少，需手动加好几笔应付

### 1.2 业务定义

费用单模块是市场管理子系统的新增组件，负责记录市场运营中的各项支出业务，支持主从结构（费用单+明细行），确认后自动联动财务模块生成应付账款。

核心业务流程：

```
创建费用单（填写主表+明细行）→ 确认 → 按明细行逐条生成 Payable → 回写 PayableID → 付款走现有付款核销流程
```

### 1.3 目标用户

| 角色 | 权限代码 | 操作范围 |
|------|---------|---------|
| 管理员 | `expense_manage` | 全部操作 |
| 工作人员 | `expense_manage` | 全部操作 |

### 1.4 设计决策记录

| 维度 | 决策 | 理由 |
|------|------|------|
| 数据结构 | 主从：费用单+明细行 | 解决业务关联、明细描述、大类归集三个痛点 |
| 应付生成 | 每条明细行→独立一条 Payable | 拆分核销，每笔费用独立付款 |
| 工作流 | 建单即确认，无草稿状态 | 简化流程，与现有应收/应付无草稿一致 |
| 导航位置 | 市场管理下拉菜单 | 费用单偏业务操作，财务管理只看生成的应付 |
| 费用大类 | 字典表维护 | 与现有字典体系一致 |
| 商户门户 | 暂不涉及 | 后续按需扩展 |

## 2. 数据模型

### 2.1 ExpenseOrder 表（费用单主表）

| 字段名 | 数据类型 | 约束 | 描述 |
|--------|----------|------|------|
| OrderID | INT | PRIMARY KEY, IDENTITY(1,1) | 主键 |
| OrderNo | NVARCHAR(20) | NOT NULL, UNIQUE | 单号，格式 EO20260416001 |
| ExpenseCategory | NVARCHAR(50) | NOT NULL | 费用大类（字典：expense_category） |
| VendorName | NVARCHAR(100) | NOT NULL | 供应商/收款方名称 |
| TotalAmount | DECIMAL(12,2) | NOT NULL | 合计金额（明细行自动求和） |
| OrderDate | DATE | NOT NULL | 费用发生日期 |
| Description | NVARCHAR(500) | NULL | 备注 |
| Status | NVARCHAR(20) | NOT NULL DEFAULT N'已确认' | 状态（已确认） |
| CreateBy | INT | NOT NULL | 创建人 UserID |
| CreateTime | DATETIME | DEFAULT GETDATE() | 创建时间 |

### 2.2 ExpenseOrderItem 表（费用单明细行）

| 字段名 | 数据类型 | 约束 | 描述 |
|--------|----------|------|------|
| ItemID | INT | PRIMARY KEY, IDENTITY(1,1) | 主键 |
| OrderID | INT | NOT NULL | 关联费用单 OrderID |
| ExpenseTypeID | INT | NOT NULL | 费用项（字典：expense_item_expend） |
| ItemDescription | NVARCHAR(200) | NULL | 明细描述（如"4月3日清扫A区"） |
| Amount | DECIMAL(12,2) | NOT NULL | 金额 |
| WorkerName | NVARCHAR(50) | NULL | 人员姓名（临时工场景） |
| WorkDate | DATE | NULL | 作业日期（临时工场景） |
| PayableID | INT | NULL | 确认后生成的应付ID，回写 |
| CreateTime | DATETIME | DEFAULT GETDATE() | 创建时间 |

### 2.3 Payable 表变更

在现有 Payable 表新增以下字段：

| 字段名 | 数据类型 | 约束 | 描述 |
|--------|----------|------|------|
| ExpenseOrderID | INT | NULL | 关联费用单 OrderID |

- `ExpenseOrderID` 允许 NULL，兼容现有手动创建的应付
- 费用单生成的应付：`ReferenceType = 'expense_order'`，`ReferenceID = OrderID`
- 费用单生成的应付：`VendorName` 取自费用单主表

### 2.4 字典新增

#### expense_category（费用大类）

| DictCode | DictName | SortOrder | 说明 |
|----------|----------|-----------|------|
| garbage_transport | 垃圾清运 | 1 | 含运费+处置费 |
| temp_labor | 临时用工 | 2 | 外雇临时工 |
| maintenance | 维修维护 | 3 | 市场设施维修 |
| other_expense | 其他支出 | 4 | 兜底项 |

#### expense_item_expend（支出费用项，新增项）

| DictCode | DictName | SortOrder | 说明 |
|----------|----------|-----------|------|
| transport_fee | 运费 | 6 | 垃圾清运-运费 |
| disposal_fee | 处置费 | 7 | 垃圾清运-处置费 |
| temp_labor_fee | 临时工费 | 8 | 临时用工-劳务费 |

> 注：现有 expense_item_expend 已有：采购/维修费/工资/水电费/税费/其他支出，排序1-5。

### 2.5 数据关系

```
ExpenseOrder 1:n ExpenseOrderItem    -- 费用单→明细行
ExpenseOrderItem 1:1 Payable         -- 明细行→应付（确认后回写 PayableID）
ExpenseOrder 1:n Payable             -- 费用单→应付（Payable.ExpenseOrderID 反向关联）
```

#### POST `/expense/create`

请求体（JSON）：

```json
{
  "expense_category": "垃圾清运",
  "vendor_name": "XX清运公司",
  "order_date": "2026-04-16",
  "description": "4月垃圾清运",
  "items": [
    {
      "expense_type_id": 6,
      "item_description": "4月垃圾运费",
      "amount": 2000.00,
      "worker_name": null,
      "work_date": null
    },
    {
      "expense_type_id": 7,
      "item_description": "4月垃圾处置费",
      "amount": 1000.00,
      "worker_name": null,
      "work_date": null
    }
  ]
}
```

业务规则：

1. `items` 至少包含 1 条明细行
2. 所有明细行 `amount` 之和 = `total_amount`（后端自动计算，不依赖前端传入）
3. 创建后自动确认，逐条明细行生成 Payable
4. Payable 字段映射：
   - `VendorName` ← 费用单的 VendorName
   - `ExpenseTypeID` ← 明细行的 ExpenseTypeID
   - `Amount` ← 明细行的 Amount
   - `DueDate` ← 费用单的 OrderDate
   - `Status` ← N'未付款'
   - `ReferenceType` ← 'expense_order'
   - `ReferenceID` ← OrderID
   - `ExpenseOrderID` ← OrderID
   - `Description` ← `{费用大类} - {费用项名} - {明细描述}`
   - `CustomerType` ← NULL（费用单的应付无客户类型）
   - `CustomerID` ← NULL
5. Payable 创建成功后，回写 `ExpenseOrderItem.PayableID`

响应：

```json
{
  "success": true,
  "message": "费用单创建成功，已生成2笔应付",
  "data": {
    "order_id": 1,
    "order_no": "EO20260416001",
    "payable_count": 2
  }
}
```

## 4. 业务规则

### 4.1 费用单创建规则

| 规则编号 | 规则描述 |
|---------|---------|
| E-01 | 费用单至少包含 1 条明细行 |
| E-02 | 明细行金额必须 > 0 |
| E-03 | TotalAmount 由后端自动计算（SUM(明细行.Amount)），不依赖前端传入 |
| E-04 | OrderNo 自动生成，格式：EO + 年月日(8位) + 序号(3位)，每日重新计数 |
| E-05 | ExpenseCategory 必须是有效的字典项（expense_category） |
| E-06 | ExpenseTypeID 必须是有效的字典项（expense_item_expend） |
| E-07 | VendorName 必填 |
| E-08 | OrderDate 必填 |

### 4.2 财务联动规则

| 规则编号 | 规则描述 |
|---------|---------|
| E-09 | 创建费用单后立即确认，状态设为 N'已确认' |
| E-10 | 确认后逐条明细行生成 Payable，整个操作在同一数据库事务内 |
| E-11 | Payable 创建成功后，回写 ExpenseOrderItem.PayableID |
| E-12 | Payable.Description 格式：`{费用大类} - {费用项名} - {明细描述}` |
| E-13 | Payable.ReferenceType = 'expense_order'，ReferenceID = OrderID |
| E-14 | 费用单生成的 Payable 无 CustomerType/CustomerID |
| E-15 | 已确认的费用单不可编辑、不可删除 |
| E-16 | 若某条明细行生成 Payable 失败，整个事务回滚 |

### 4.3 临时工特殊规则

| 规则编号 | 规则描述 |
|---------|---------|
| E-17 | WorkerName 和 WorkDate 为可选字段，仅临时工场景需要 |
| E-18 | 同一费用单可包含多条临时工明细行（不同人、不同日期） |
| E-19 | 临时工明细行的 ItemDescription 建议格式：`{WorkDate} {WorkerName} {工作内容}` |

- 保存按钮：点击即创建+确认+生成应付

## 9. 数据库迁移脚本

```sql
USE hf_metalmarket;
GO

-- 1. 创建费用单主表
CREATE TABLE ExpenseOrder (
    OrderID INT PRIMARY KEY IDENTITY(1,1),
    OrderNo NVARCHAR(20) NOT NULL UNIQUE,
    ExpenseCategory NVARCHAR(50) NOT NULL,
    VendorName NVARCHAR(100) NOT NULL,
    TotalAmount DECIMAL(12,2) NOT NULL,
    OrderDate DATE NOT NULL,
    Description NVARCHAR(500) NULL,
    Status NVARCHAR(20) NOT NULL DEFAULT N'已确认',
    CreateBy INT NOT NULL,
    CreateTime DATETIME DEFAULT GETDATE()
);

-- 2. 创建费用单明细表
CREATE TABLE ExpenseOrderItem (
    ItemID INT PRIMARY KEY IDENTITY(1,1),
    OrderID INT NOT NULL,
    ExpenseTypeID INT NOT NULL,
    ItemDescription NVARCHAR(200) NULL,
    Amount DECIMAL(12,2) NOT NULL,
    WorkerName NVARCHAR(50) NULL,
    WorkDate DATE NULL,
    PayableID INT NULL,
    CreateTime DATETIME DEFAULT GETDATE()
);

-- 3. Payable 表新增 ExpenseOrderID 字段
ALTER TABLE Payable ADD ExpenseOrderID INT NULL;

-- 4. 字典数据：费用大类
INSERT INTO Sys_Dictionary (DictType, DictCode, DictName, SortOrder, IsActive)
VALUES
(N'expense_category', N'garbage_transport', N'垃圾清运', 1, 1),
(N'expense_category', N'temp_labor', N'临时用工', 2, 1),
(N'expense_category', N'maintenance', N'维修维护', 3, 1),
(N'expense_category', N'other_expense', N'其他支出', 4, 1);

-- 5. 字典数据：支出费用项（新增项）
INSERT INTO Sys_Dictionary (DictType, DictCode, DictName, SortOrder, IsActive)
VALUES
(N'expense_item_expend', N'transport_fee', N'运费', 6, 1),
(N'expense_item_expend', N'disposal_fee', N'处置费', 7, 1),
(N'expense_item_expend', N'temp_labor_fee', N'临时工费', 8, 1);

-- 6. 权限数据
INSERT INTO Permission (PermissionName, PermissionCode, Description, ModuleName, IsActive)
VALUES (N'费用单管理', N'expense_manage', N'管理费用单的创建、查看', N'业务管理', 1);

-- 7. 给管理员和工作人员角色赋权
DECLARE @AdminRoleID INT = (SELECT RoleID FROM Role WHERE RoleCode = 'admin');
DECLARE @StaffRoleID INT = (SELECT RoleID FROM Role WHERE RoleCode = 'staff');
DECLARE @PermID INT = (SELECT PermissionID FROM Permission WHERE PermissionCode = 'expense_manage');

IF @AdminRoleID IS NOT NULL AND @PermID IS NOT NULL
    INSERT INTO RolePermission (RoleID, PermissionID) VALUES (@AdminRoleID, @PermID);
IF @StaffRoleID IS NOT NULL AND @PermID IS NOT NULL
    INSERT INTO RolePermission (RoleID, PermissionID) VALUES (@StaffRoleID, @PermID);
GO
```
