## 快速开始

### 基本使用

当你有新的功能需求时，只需要一句话：

```
运行工作流 workflows/project-development.yaml
需求：[你的需求描述]
```

示例：
```
运行工作流 workflows/project-development.yaml
需求：创建一个商户登录页面，包含用户名、密码输入框和登录按钮
```

### 选择合适的工作流

| 工作流 | 适用场景 | 特点 |
|--------|----------|------|
| `project-development.yaml` | 完整功能开发 | 包含完整的需求分析、计划、子智能体开发流程 |
| `quick-feature.yaml` | 小型功能、bug修复 | 简化流程，快速实现 |

## 在新项目中使用

### 步骤 1：准备工作

1. 确保你的项目中有 `.trae/skills/` 目录，包含所有需要的技能文件
2. 检查是否有以下必需技能：
   - `brainstorming` - 需求分析
   - `writing-plans` - 计划编写
   - `subagent-driven-development` - 子智能体开发
   - `executing-plans` - 计划执行
   - `test-driven-development` - 测试驱动开发
   - `systematic-debugging` - 系统化调试
   - `requesting-code-review` - 代码审查
   - `receiving-code-review` - 处理审查反馈
   - `verification-before-completion` - 完成前验证
   - `finishing-a-development-branch` - 收尾开发

3. 确保有角色定义文件：
   - `project-manager.md` - 项目经理角色

### 步骤 2：复制工作流文件

将以下文件复制到新项目的 `workflows/` 目录：
```
workflows/
├── project-development.yaml
├── quick-feature.yaml
├── README.md
└── 使用教程.md
```

### 步骤 3：配置工作流（可选）

根据项目需要，编辑 YAML 文件中的 `agents_dir` 路径：

```yaml
agents_dir: ".trae/skills"  # 修改为你实际的技能目录路径
```

### 步骤 4：开始使用

在新项目中，直接触发工作流：

```
运行工作流 workflows/project-development.yaml
需求：[描述你的项目需求]
```

## 工作流详解

### project-development.yaml 完整流程

#### 阶段 1：需求分析（brainstorming）
- **目标**：明确需求，设计规格
- **流程**：
  1. 探索项目上下文
  2. 澄清问题（每次一个问题）
  3. 提出 2-3 个方案
  4. 展示设计
  5. 自检规格文档
  6. 用户审查确认
- **输出**：`docs/superpowers/specs/YYYY-MM-DD-xxx-design.md`

#### 阶段 2：计划编写（writing-plans）
- **目标**：将规格拆分为可执行步骤
- **流程**：
  1. 范围检查
  2. 设计文件结构
  3. 任务分解（小步骤）
  4. 自检
- **输出**：`docs/superpowers/plans/YYYY-MM-DD-xxx.md`

#### 阶段 3：用户确认
- 向用户展示完整计划
- 等待用户批准后继续

#### 阶段 4：执行开发（subagent-driven-development）
- **目标**：按计划实现功能
- **流程**：
  每个任务：
  1. 分派实现子智能体
  2. 子智能体遵循 TDD 开发
  3. 自审
  4. 规格合规性审查（第 1 阶段）
  5. 代码质量审查（第 2 阶段）
  6. 验证完成
  7. 如有独立任务，并行执行
  8. 如遇 bug，系统化调试

#### 阶段 5：收尾（finishing-a-development-branch）
- 最终验证所有功能
- 向用户展示成果

### quick-feature.yaml 简化流程

适合场景：
- 小型功能（< 1 天工作量）
- 简单的 bug 修复
- 快速原型验证

流程简化：
- 使用 `executing-plans` 而非子智能体
- 在当前会话中直接执行
- 减少审查环节，提高速度

## 技能集成说明

### 必需技能

| 技能 | 用途 | 在工作流中的位置 |
|------|------|------------------|
| `brainstorming` | 需求分析 → 规格设计 | 阶段 1 |
| `writing-plans` | 规格 → 执行计划 | 阶段 2 |
| `subagent-driven-development` | 子智能体分任务开发 | 阶段 4（完整版） |
| `executing-plans` | 同会话执行计划 | 阶段 4（简化版） |
| `test-driven-development` | 测试驱动开发，先写测试再写代码 | 子智能体执行 |
| `systematic-debugging` | 定位、分析、修复 bug | 遇到问题时 |
| `requesting-code-review` | 申请代码审查 | 任务完成后 |
| `receiving-code-review` | 处理审查反馈 | 收到审查后 |
| `verification-before-completion` | 完成前验证证据 | 每个任务完成前 |
| `dispatching-parallel-agents` | 并行分派独立任务 | 有独立任务时 |
| `finishing-a-development-branch` | 收尾，合并准备 | 阶段 5 |

### 技能使用要点

#### 1. brainstorming
- **HARD-GATE**：在展示设计并获得用户批准前，**不调用任何实现技能**
- 每次只问一个问题
- 提供 2-3 个方案供选择
- 严格按检查清单执行

#### 2. writing-plans
- 假设工程师**零上下文**
- 每个任务要小到 2-5 分钟可完成
- **禁止占位符**：没有"待定"、"TODO"
- 包含完整的代码示例和命令

#### 3. test-driven-development
- **铁律**：没有失败的测试，不写生产代码
- 红 → 绿 → 重构：
  1. 红灯：写失败的测试
  2. 绿灯：写最少代码让测试通过
  3. 重构：清理代码，保持绿灯

#### 4. systematic-debugging
- **铁律**：不做根因调查，不许提修复方案
- 4 个阶段：
  1. 根因调查
  2. 模式分析
  3. 假设与验证
  4. 实施修复

#### 5. verification-before-completion
- **铁律**：没有新鲜验证证据，不许宣称完成
- 始终用证据支撑结论

## 常见问题

### Q1: 如何选择工作流？

**A:**
- 新项目、大功能 → `project-development.yaml`
- 小功能、bug修复 → `quick-feature.yaml`

### Q2: 工作流可以自定义吗？

**A:** 可以！复制 YAML 文件，修改 `steps` 部分调整流程。

### Q3: 可以添加新角色吗？

**A:** 可以！在 `.trae/skills/` 目录添加新的角色定义文件，然后在 YAML 的 `role` 字段中引用。

### Q4: 如何回退到某个阶段？

**A:** 工作流是线性的，如需调整，直接回到该阶段重新开始即可。例如：
```
让我们回到 brainstorming 阶段，重新讨论需求
```

### Q5: 工作流执行中断怎么办？

**A:**
1. 检查错误信息
2. 确认是环境问题还是流程问题
3. 环境问题 → 修复环境后继续
4. 流程问题 → 回到上一个确认点重新开始

### Q6: 如何将此系统推广到其他项目？

**A:** 遵循以下步骤：
1. 复制 `workflows/` 目录到新项目
2. 复制 `.trae/skills/` 目录（或确保新项目有相同技能）
3. 如需要，调整 YAML 中的 `agents_dir` 路径
4. 开始使用！

### Q7: 工作流中的用户确认关卡可以跳过吗？

**A:** **不建议跳过**。这些关卡是质量保证的重要环节。但你可以创建自定义工作流，删除确认步骤。

### Q8: 如何并行处理多个任务？

**A:**
1. 使用 `project-development.yaml`
2. 任务设计时明确哪些是独立的
3. 工作流会自动使用 `dispatching-parallel-agents` 并行执行

## 示例使用场景

### 场景 1：新功能开发

```
用户：运行工作流 workflows/project-development.yaml
      需求：添加一个合同导出功能，支持导出为 Word 文档

工作流执行：
1. brainstorming → 需求分析
2. writing-plans → 创建计划
3. 用户确认计划
4. subagent-driven-development → 开发实现
5. finishing-a-development-branch → 收尾
```

### 场景 2：快速修复 Bug

```
用户：运行工作流 workflows/quick-feature.yaml
      需求：修复商户列表页面搜索不工作的 bug

工作流执行：
1. brainstorming（简化）→ 分析问题
2. writing-plans → 创建修复计划
3. 用户确认
4. executing-plans → 快速修复
5. finishing-a-development-branch → 收尾
```

### 场景 3：新项目从零开始

```
用户：运行工作流 workflows/project-development.yaml
      需求：创建一个完整的商户管理系统，包含登录、列表、增删改查功能

工作流执行：
（完整流程，会拆解为多个子任务）
```

## 最佳实践

1. **从小开始**：先用 `quick-feature.yaml` 尝试小功能，熟悉流程
2. **详细描述需求**：需求越清晰，brainstorming 阶段越快
3. **及时反馈**：每个确认关卡及时给出反馈，不要积压
4. **信任流程**：每个技能都经过验证，按流程走能保证质量
5. **保存文档**：规格和计划文档要保留，方便后续维护
